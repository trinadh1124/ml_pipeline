Problem Statement

Business context

A telecom company is losing customers. The churn rate across the customer base is 26.5% — more than one in four customers leaves. Each lost customer costs approximately $1,500 in new customer acquisition to replace. The business wants to predict which customers are likely to churn so the retention team can intervene with targeted offers.

ML problem definition

InputOne row per customer — 20 features: demographics, services, contract, billing
Outputchurn = 1 (will leave) or churn = 0 (will stay), plus probability
Target variablechurn — binary. Positive class = customer churned
Class distribution~23.4% churn in the historical batch, ~23.3% churn in the stable current batch, and a deliberately shifted 51.2% churn in the stress batch
Primary metricF1 on the minority class (churners) in the historical hold-out set

Pipeline Story

This pipeline teaches two monitoring outcomes clearly. First, a normal post-deployment batch behaves like the historical baseline, so monitoring stays calm. Second, a separate stress batch is intentionally shifted toward high-risk customer segments, so drift is easy to detect and retraining has a believable reason.

Phase 1 — Training
Historical Batch
5,300 customers. Churn rate: 23.4%. This is the data available at deployment time and used for train/hold-out evaluation.
Phase 2 — Stable Monitoring
Current Batch
943 customers. Churn rate: 23.3%. Used to show a normal post-deployment batch where no retraining is needed.
Phase 3 — Stress Monitoring
Current Stress Batch
800 customers. Churn rate: 51.2%. Built from high-risk natural segments so drift is visible and retraining is recommended.

What makes the stress batch realistic? It uses existing columns only: contract, internet_service, payment_method, and optionally short tenure_months. No fake columns are invented. The batch is simply enriched for naturally high-risk customers.

Step 3 — main.py (12-Stage Pipeline)

Pipeline Stages

Stage 0
Initialisation
Set seed (42), create directories, configure MLflow tracking URI and experiment.
Stage 1
Data Loading
Load data/train.csv (5,300 historical), data/current.csv (943 stable current), and data/current_stress.csv (800 stress current). All are tracked by DVC.
Stage 2
Data Validation (Pandera)
Declarative schema validates types, ranges, categorical domains, and nullable columns. Replaces Module 1's manual if-else checks.
Stage 3
Feature Engineering
One-hot encodes nominals (contract, internet service, payment method). Fits imputations on the training fold only, then saves the pipeline to feature_pipeline.json so hold-out and serving use the same transformation without leakage.
Stage 4
Train/Test Split
80/20 stratified split of the historical batch only. Hold-out 20% for pre-deployment validation.
Stage 5
Model Training (MLflow tracked)
Train the tuned production estimator selected by model_selection.py. In the current project that winner is XGBoost, and its parameters plus model artefact are logged to MLflow automatically.
Stage 6
Pre-Deployment Evaluation
Reports both the default threshold (0.50) and the deployment threshold chosen from the business cost model with a precision guardrail. Slice analysis by contract type reveals performance gaps.
Stage 7
Deployment Gates
Hard thresholds: F1 ≥ 0.48, Recall ≥ 0.55, AUC-PR ≥ 0.40, AUC-ROC ≥ 0.70. If any gate fails, pipeline stops.
Stage 8
MLflow Model Logging
Model artefact with inferred signature logged. Feature pipeline JSON also logged as artefact.
Stage 9
Model Registry (MLflow)
Registered as telco-churn-production. Promoted: None → Staging → Production. Previous versions auto-archived.
Stage 10
Drift Detection (Evidently)
Compares the historical batch to whichever post-deployment batch you choose. Stable monitoring writes logs/drift_report_current.html; stress monitoring writes logs/drift_report_stress.html.
Stage 11
Batch Scoring
Scores either the stable current batch or the stress batch with the saved production threshold. This lets learners compare a normal monitoring outcome versus a retraining-trigger outcome using the same deployed model.
Stage 12
Lineage Recording
Full provenance: data checksums, model version, MLflow run ID, Python version, git commit.

retrain.py now supports two learner-friendly monitoring outcomes. Use --batch current to show a stable post-deployment batch where the answer is "keep monitoring". Use --batch stress to show the shifted batch where retraining is recommended.

Model Performance Monitoring vs Data Drift

These are two different checks learners should keep separate. In this project, the stable current batch shows how monitoring can stay calm, while the stress batch shows how drift can trigger action.

ConceptData DriftPerformance Monitoring
What it detectsInput feature distributions changedModel outputs are wrong
Requires labels?No — compares feature distributions onlyYes — needs ground truth (actual churn)
When to checkImmediately on new data arrivalAfter ground truth labels become available
Examplecurrent_stress.csv shifts toward month-to-month, fiber optic, electronic-check, short-tenure customersThe deployed model performs differently once labels are available
Tool in Module 2Evidently DataDriftPresetMLflow metric logging + deployment gates

Why both are needed: Data drift tells you the model might be facing a new situation. Performance monitoring tells you whether that new situation is actually hurting outcomes once labels arrive. In the guide, current.csv demonstrates "monitor only" and current_stress.csv demonstrates "retraining recommended".

The monitoring loop when ground truth is delayed

In telecom churn, a customer may take 30-90 days to actually churn after the model predicts they will. During this delay:

  • Day 0: Model scores either current.csv or current_stress.csv
  • Day 0: Evidently drift report checks whether the selected batch has shifted
  • Day 30-90: Ground truth labels arrive (customer actually churned or stayed)
  • Day 30-90: F1/Recall/Precision computed against ground truth — confirms degradation
  • Decision: Retrain based on drift (proactive) or degradation (reactive)?

When and How to Retrain

Three retraining strategies, each with different trade-offs in cost, risk, and complexity:

StrategyHow it worksCostRiskWhen to use
Scheduled Retrain on a fixed cadence (weekly, monthly) Medium — runs even when not needed Low — model stays fresh Stable environments with gradual drift
Triggered Retrain only when drift or degradation is detected Low — runs only when needed Medium — depends on detection quality Environments with sudden shifts (our telecom case)
Continuous Online learning — model updates on every new batch High — always running High — can chase noise High-frequency data, streaming scenarios

Our pipeline uses triggered retraining. The intended learner story is simple: the stable batch should not retrain, the stress batch should retrain, and mixed is the recommended response because it keeps the historical baseline while adapting to the shifted segment.

Threshold note for retraining: the project now keeps one fixed learner-facing threshold of 0.20 across hold-out evaluation, batch scoring, and retraining. This makes the deployment story easier to follow.

Current production thresholding results

The latest pipeline evaluates the tuned production model on the historical hold-out set, then applies the same fixed 0.20 threshold to whichever post-deployment batch is being monitored.

BatchRowsChurn RateExpected learning outcome

The key teaching improvement is the split design. Learners can now see both outcomes clearly: one monitored batch that stays stable and one monitored batch that justifies retraining, without inventing fake features or unrealistic columns.

Stress monitoring guardrail: main.py --batch stress is monitor-only. It scores the shifted batch and generates drift evidence, but it does not replace the stable production model. Promotion happens only through retraining if the new candidate passes gates.

Pandera — Data Validation

Pandera replaces Module 1's manual schema checks with a declarative schema.

# What Module 1 did:
if df.isnull().sum().sum() > threshold:
    raise ValueError("Too many nulls")
if not set(df.columns) >= expected_cols:
    raise ValueError("Missing columns")

# What Module 2 does:
import pandera as pa
schema = pa.DataFrameSchema({
    "tenure_months": pa.Column(int, pa.Check.in_range(0, 72)),
    "contract":      pa.Column(str, pa.Check.isin([
        "Month-to-month", "One year", "Two year"
    ])),
    "monthly_charges": pa.Column(float, pa.Check.in_range(0, 200)),
})
validated_df = schema.validate(df)  # raises on failure

DVC — Data Versioning

DVC tracks dataset versions alongside code in Git. When you change the training data, DVC records the new version.

# Initialise

# Track data files
dvc add data/train.csv data/current.csv data/current_stress.csv
git add data/train.csv.dvc data/current.csv.dvc data/current_stress.csv.dvc

# Push data to remote
dvc push

# Tag a pipeline version
git tag -a v1.0 -m "Initial model training"

# Restore any previous data version
git checkout v1.0 -- data/train.csv.dvc
dvc checkout

Evidently — Drift Detection

Evidently replaces Module 1's manual mean/std comparison with proper statistical tests.

# What Module 1 did:
shift = abs(train_mean - current_mean) / train_mean
if shift > 0.10:
    print(f"DRIFT: {feature} shifted {shift:.1%}")

# What Module 2 does:
from evidently.legacy.report import Report
from evidently.legacy.metric_preset import DataDriftPreset

report = Report(metrics=[DataDriftPreset()])
report.run(reference_data=train_df, current_data=current_df)
report.save_html("logs/drift_report_current.html")
# Save a separate report for the stress batch when needed

SHAP — Model Interpretability

XGBoost feature importance tells you which features the model uses most. SHAP answers the harder question: "Why was this specific customer flagged as a churner?"

→
Module 2

from src.models.explain import explain_model

Business value: The retention team no longer gets a bare 0/1 prediction. They get: "This customer is likely to churn because their contract is month-to-month (+0.18 toward churn) and tenure is only 3 months (+0.12), despite having tech support (-0.05 away from churn)." Actionable, not opaque.

Optuna — Hyperparameter Tuning

Module 1 used fixed hyperparameters. The reexperiment strategy in retrain.py searched a crude 4-config grid. Now Optuna runs Bayesian optimization — learning from previous trials to focus on promising regions of the search space for the top candidates from model selection.

# In model_selection.py — tune the top candidates, then persist the winner
from src.models.tuning import tune_model

best_params, best_f1 = tune_model(
    model_name="XGBoost",
    X_train=X_train, y_train=y_train,
    n_trials=50,    # 50 Bayesian trials
)
# Searches model-specific spaces and writes models/best_params.json
# main.py then reads that file and deploys the selected config
ApproachTrialsIntelligenceWhen to use
Grid searchAll combosNone — brute forceTiny search spaces (<50 combos)
Random searchRandom sampleUniform samplingFirst pass exploration
Optuna (TPE)BayesianLearns from prior trialsProduction tuning — efficient and adaptive

Why 50 trials, not 300? TPE (Tree-structured Parzen Estimator) converges fast. On this dataset size, 50 trials gives a strong trade-off between quality and runtime. More trials yield diminishing returns.

DVC Pipeline — Reproducible Stages

Beyond dvc add (data tracking), DVC also defines reproducible pipeline stages via dvc.yaml. Each stage declares its inputs (deps), outputs (outs), and command. DVC caches results and skips unchanged stages.

# dvc.yaml defines 4 stages:
#   validate → features → train → drift

# Run the full pipeline (skips unchanged stages)
dvc repro

# Run only up to training
dvc repro train

# View the DAG
dvc dag
# Output:
#   validate → features → train → drift

# Compare metrics between commits
dvc metrics show
dvc metrics diff

# Compare parameter changes
dvc params diff
StageDependenciesOutputsWhat it does
validatedata/*.csv, validate.pylogs/.validate_okPandera schema validation on both datasets
featurestrain.csv, build_features.pyfeature_pipeline.jsonBuild and save feature pipeline
trainAll source moduleslogs/metrics.json, SHAP plotsFull 12-stage pipeline with metric export
driftdata/*.csv, drift.pydrift_report_current.html / drift_report_stress.htmlEvidently statistical drift detection for the selected monitoring batch

Why this matters: dvc repro guarantees that anyone on the team gets the same results from the same data + code. Change the training data? DVC reruns features → train → drift. Change only drift.py? Only drift reruns. This is the foundation for reproducible ML experiments.
